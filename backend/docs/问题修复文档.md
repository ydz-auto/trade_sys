## 问题 1: Kafka 集群 ID 不匹配

### 错误日志

```
2026-05-17 11:05:23 WARNING [aiokafka.consumer.group_coordinator] Marking the coordinator dead (node 1)for group tradeagent-execution.
2026-05-17 11:05:24 INFO [aiokafka.consumer.group_coordinator] Discovered coordinator 1 for group tradeagent-execution
2026-05-17 11:05:24 INFO [aiokafka.consumer.group_coordinator] (Re-)joining group tradeagent-execution
2026-05-17 11:05:24 WARNING [aiokafka.consumer.group_coordinator] Marking the coordinator dead (node 1)for group tradeagent-execution.
```

```
2026-05-17 11:05:37 WARNING [aiokafka.producer.sender] Got error produce response on topic-partition TopicPartition(topic='tradeagent.events', partition=1), retrying. Error: <class 'aiokafka.errors.NotLeaderForPartitionError'>
```

```
2026-05-17 11:05:46 ERROR [aiokafka.consumer.fetcher] Failed fetch offsets from 1: [Error 6] NotLeaderForPartitionError: TopicPartition(topic='tradeagent.events', partition=1)
```

### 根本原因

**ZooKeeper 模式的架构缺陷**：

1. **Topic ID 不一致**
   - ZooKeeper 和 Kafka Broker 分别维护元数据
   - 手动重启 Kafka 时，两边的 Topic ID 可能不同步
   - Kafka 日志显示：`Topic ID in memory: 2fP-KhvDR1C_yJo1UuDplQ does not match the topic ID received: 4SpQmUI3Qme59jFsU-mR-g`

2. **Controller 故障转移问题**
   - Controller 选举依赖 ZooKeeper
   - ZooKeeper 连接不稳定导致 coordinator 死亡循环

3. **Metadata Cache 不一致**
   - Broker cache 可能与 ZooKeeper 不同步
   - 导致 `NotLeaderForPartitionError`

### 解决方案

**迁移到 KRaft 模式**（Apache Kafka 3.x 推荐模式）

#### 1. 架构对比

```
ZooKeeper 模式:                    KRaft 模式:
┌─────────────┐                   ┌─────────────┐
│  ZooKeeper  │                   │             │
│  (外部系统)  │                   │   无需      │
└──────┬──────┘                   │  ZooKeeper  │
       │                          │             │
       ▼                          └─────────────┘
┌─────────────┐                         
│   Kafka     │                   ┌─────────────┐
│   Broker    │                   │   Kafka     │
│             │                   │   Broker +  │
│  (依赖 ZK)  │                   │   Controller│
└─────────────┘                   │  (一体化)   │
                                  └─────────────┘
```

#### 2. 配置变更

**docker-compose.yml**:

```yaml
services:
  kafka:
    image: confluentinc/cp-kafka:7.5.0
    environment:
      # KRaft 模式核心配置
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_NODE_ID: 1
      KAFKA_CONTROLLER_QUORUM_VOTERS: 1@kafka:9093
      KAFKA_CONTROLLER_LISTENER_NAMES: CONTROLLER
      
      # 监听器配置
      KAFKA_LISTENERS: PLAINTEXT://:9092,PLAINTEXT_INTERNAL://:29092,CONTROLLER://:9093
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092,PLAINTEXT_INTERNAL://kafka:29092
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT,PLAINTEXT_INTERNAL:PLAINTEXT
      
      # KRaft 元数据目录
      KAFKA_METADATA_LOG_DIR: /var/lib/kafka/data/kraft-metadata
      KAFKA_LOG_DIRS: /var/lib/kafka/data/kraft-logs
      
      # 固定集群 ID（避免不一致）
      CLUSTER_ID: MkU3OEVBNTcwNTJENDM2Qk
```

**移除的配置**:
- `KAFKA_ZOOKEEPER_CONNECT`
- `KAFKA_BROKER_ID`（替换为 `KAFKA_NODE_ID`）
- ZooKeeper service

### 效果对比

| 问题 | ZooKeeper 模式 | KRaft 模式 |
|------|---------------|-----------|
| **Topic ID 一致性** | ❌ ZooKeeper 和 Broker 元数据可能不同步 | ✅ 统一的 metadata log |
| **Controller Failover** | ❌ 需要从 ZooKeeper 选举，可能失败 | ✅ 内置 Raft quorum |
| **Metadata Cache** | ❌ 多个来源可能不一致 | ✅ 单一数据源 |
| **重启恢复** | ❌ 可能需要手动修复 | ✅ 自动恢复 |
| **集群 ID 不匹配** | ❌ 经常出现 | ✅ 固定 ID，永不出错 |

## 问题 2: Correlation 模块代码错误

### 错误日志

```
2026-05-17 11:03:15 ERROR [correlation] Analysis failed: ETH 4h - 'CorrelationResult' object has no attribute 'get'
```

### 根本原因

在 `research/correlation/univariate_analysis.py` 中定义了 `CorrelationResult` dataclass：

```python
@dataclass
class CorrelationResult:
    feature: str
    target: str
    pearson: float
    spearman: float
    kendall: float
    p_value_pearson: float
    p_value_spearman: float
    p_value_kendall: float
    sample_size: int
```

但在 `research/correlation/analyzer.py` 中，`_univariate_analysis()` 方法直接返回了 dataclass 对象，而后续代码（如 `scoring.py`）尝试调用 `.get()` 方法（字典方法）：

```python
# 错误的代码
results[feature] = {
    "correlation": self.univariate.correlation_analysis(...),  # 返回 dataclass
    "mutual_info": self.univariate.mutual_information(...),   # 返回 dataclass
}

# 后续代码尝试使用字典方法
result.get("correlation")  # ❌ dataclass 没有 get 方法
```

### 解决方案

修改 `research/correlation/analyzer.py`，将 dataclass 对象转换为字典：

```python
async def _univariate_analysis(self, feature_matrix: FeatureMatrix) -> Dict[str, Any]:
    """单变量分析"""
    results = {}
    
    feature_cols = [c for c in feature_matrix.data.columns if c != feature_matrix.target_col]
    
    for feature in feature_cols:
        corr_result = self.univariate.correlation_analysis(
            feature_matrix.data, feature, feature_matrix.target_col
        )
        mi_result = self.univariate.mutual_information(
            feature_matrix.data, feature, feature_matrix.target_col
        )
        granger_result = self.univariate.granger_causality(
            feature_matrix.data, feature, feature_matrix.target_col
        )
        lag_result = self.univariate.lag_correlation(
            feature_matrix.data, feature, feature_matrix.target_col,
            max_lag=max(self.config.lag_windows)
        )
        
        # 转换 dataclass 为字典
        results[feature] = {
            "correlation": corr_result.to_dict() if hasattr(corr_result, 'to_dict') else corr_result,
            "mutual_info": mi_result.to_dict() if hasattr(mi_result, 'to_dict') else mi_result,
            "granger": {k: v.to_dict() if hasattr(v, 'to_dict') else v for k, v in granger_result.items()} if isinstance(granger_result, dict) else granger_result,
            "lag_correlation": lag_result.to_dict() if hasattr(lag_result, 'to_dict') else lag_result,
        }
    
    return results
```

同样的修复应用到 `_multivariate_analysis()` 方法。

## 经验总结

### 1. Kafka KRaft 模式优势

- **简化架构**: 无需 ZooKeeper，减少运维复杂度
- **提高可靠性**: 统一的元数据管理，避免不一致问题
- **更快恢复**: Controller 故障转移更快
- **未来趋势**: Apache Kafka 4.0 将完全移除 ZooKeeper 支持

### 2. 开发环境最佳实践

- **使用固定集群 ID**: 避免重启后的 ID 不匹配问题
- **定期清理数据**: 开发环境可以定期重置 Kafka 数据
- **监控健康状态**: 通过 Kafka UI 监控集群状态

### 3. 代码规范

- **类型一致性**: 返回值类型要统一（要么都是 dict，要么都是 dataclass）
- **接口设计**: 如果需要 dict 操作，应该在返回前转换
- **错误处理**: 添加类型检查和转换逻辑
