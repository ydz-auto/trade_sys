# P0.1: Deterministic Replay Engine - 确定性回放引擎

> **核心理念**: 如果 Replay ≠ Live，那是 Runtime Kernel 的 bug，不是 Feature 的问题
>
> **目标**: 绝对一致性验证器，验证你的系统是否已经可以上 Live

---

## 🎯 核心任务

### 验证矩阵（Replay ≡ Live）

| 验证项 | 验证内容 | 严重程度 |
|--------|----------|----------|
| **Event Ordering** | 事件顺序完全一致 | 🔴 致命 |
| **Feature Availability** | 特征可见性完全一致 | 🔴 致命 |

## 🔧 核心组件

### 1. Event Log 系统

**职责**:
- 在 Live 模式下记录所有事件
- 在 Replay 模式下回放完全相同的事件序列

**记录内容**:
```python
LoggedEvent = {
    event_id: str,
    event_type: str,
    event_time_ms: int,
    available_time_ms: int,
    processing_time_ms: int,
    sequence_number: int,
    payload: FrozenDict,
    source: str,
    hash: str,
}
```

### 2. State Capture 系统

**职责**:
- 定期捕获系统完整状态
- 在关键状态转换点强制捕获

**捕获点**:
- 每个事件处理前/后
- 每个 K 线关闭时
- 每个订单提交/成交时
- 每个组合变化时

### 3. Determinism Validator

**职责**:
- 验证两次回放是否完全一致
- 验证 Replay vs Live 是否完全一致
- 定位不一致的精确位置和原因

## 📊 工作流程

### Record Mode（记录模式 - Live）

```
1. 启动 Recording
2. Runtime 正常处理事件
3. Event Log 记录每个事件
4. State Capture 在每个关键点记录状态
5. 停止 Recording，保存 Log + Snapshots
```

### Replay Mode（回放模式）

```
1. 加载 Recorded Event Log
2. 切换 Runtime 到 REPLAY 模式
3. Event Playback Controller 逐个按顺序播放事件
4. 在完全相同的时间点捕获 State
5. 与 Live 记录的 State 进行逐字节比对
```

### Verification Mode（验证模式）

```
1. 加载 Live Record + Replay Result
2. 逐事件对比状态
3. 生成详细的不一致报告
4. 如果 100% 一致 → 系统已可上 Live
```

## ✅ 验收标准

### 黄金标准

| 指标 | 要求 |
|------|------|
| **Event Sequence** | 100% 一致 |
| **Timing** | 100% 一致（时钟前进完全同步）|
| **Feature Value** | 100% 一致 |
| **Signal Output** | 100% 一致 |
| **Order/Trade** | 100% 一致 |
| **Portfolio Value** | 100% 一致 |

### 错误分级

| 级别 | 描述 | 处理 |
|------|------|------|
| 🔴 **Fatal** | 时钟、顺序、可见性错误 | 系统不可用，必须修复 |
| 🟠 **Critical** | 特征、信号、组合错误 | 严重问题，必须修复 |
| 🟡 **Warning** | 非关键差异 | 可接受，但应调查 |
| 🟢 **OK** | 完全一致 | ✅ 通过 |
