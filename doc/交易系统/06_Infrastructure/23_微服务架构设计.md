# TradeAgent 微服务架构设计文档

---

# 🧠 一、设计目标

```
从"模块化单体"演进到"微服务架构"
- 支持高可用、高并发、水平扩展
- 各服务独立部署、独立升级
- 适合生产环境的多交易对（BTC/ETH）+ 多策略
```

---

# 🏗️ 二、微服务划分总览

| 服务 | 功能 | 技术栈 | 数据库/存储 |
|------|------|--------|-------------|
| **api-gateway** | 前端统一接口、路由、负载均衡 | FastAPI / Nginx | - |
| **auth-service** | 用户认证、KYC、权限管理 | Python + FastAPI | PostgreSQL |
| **data-service** | 数据采集 + ETL（行情、ETF、新闻、宏观） | Python + FastAPI | ClickHouse / Redis |
| **event-service** | 语义理解、数据 → 事件转换 | Python + FastAPI + LLM | Redis |
| **fusion-service** | 多源信息融合、置信度计算 | Python + FastAPI | Redis / ClickHouse |
| **feature-service** | 特征计算（收益率、波动率、动量、资金流、情绪） | Python + FastAPI | ClickHouse |
| **factor-service** | 因子计算（Trend、Flow、Sentiment、Macro、Behavioral） | Python + FastAPI | ClickHouse |
| **risk-service** | 风险指数计算、杠杆控制、止损止盈 | Python + FastAPI | PostgreSQL / Redis |
| **decision-service** | 决策引擎、多资产交易逻辑、信号输出 | Python + FastAPI | PostgreSQL / Redis |
| **position-service** | 仓位计算、动态调整 | Python + FastAPI | PostgreSQL / Redis |
| **execution-service** | 模拟执行 / 实盘下单隔离 | Python + FastAPI | PostgreSQL / Redis |
| **llm-service** | LLM 情绪分析、爬虫增强、策略解释 | Python + FastAPI + LLM | Redis |
| **feedback-service** | 反馈闭环、PnL分析、策略调整 | Python + FastAPI | PostgreSQL / ClickHouse |
| **state-service** | 全局状态管理（仓位/账户/风险/策略状态） | Python + FastAPI | Redis / PostgreSQL |
| **config-service** | 配置中心、因子权重、风险参数、版本管理 | Python + FastAPI | PostgreSQL |
| **monitor-service** | 系统监控、健康检查、告警 | Python + FastAPI | Redis |
| **notification-service** | 邮件/短信/推送通知 | Python + FastAPI | Redis |
| **regime-service** | 市场状态判断、趋势识别 | Python + FastAPI | ClickHouse / Redis |
| **portfolio-service** | 组合风险管理、相关性分析 | Python + FastAPI | PostgreSQL / Redis |
| **control-service** | 系统控制、暂停/恢复/紧急止损 | Python + FastAPI | Redis / PostgreSQL |

> 服务间通信：**Kafka / RabbitMQ**（异步数据流）+ **Redis**（缓存和实时状态）

**总计：20个微服务**（新增 event-service + fusion-service）

# 🔄 四、服务通信模式

## 4.1 通信类型总览

| 通信类型 | 技术选型 | 说明 | 适用场景 |
|----------|----------|------|----------|
| **异步数据流** | Kafka / Redis PubSub | 消息队列、事件驱动 | 行情数据、订单事件、信号传递 |
| **实时查询** | HTTP / gRPC | 请求-响应、同步调用 | 状态查询、配置获取 |
| **实时推送** | WebSocket + Redis | 服务器主动推送 | 前端行情、订单状态推送 |
| **任务队列** | Celery + Redis | 异步任务、定时计算 | 批量计算、报告生成 |

## 4.2 异步数据流（Kafka / Redis）

```
┌─────────────┐    Publish    ┌─────────────────┐
│ data-service │ ───────────▶ │ Kafka/RabbitMQ   │
│             │              │ topic: raw_data  │
└─────────────┘              └────────┬──────────┘
                                      │
        ┌─────────────────────────────┼─────────────────────────────┐
        │                             │                             │
        ▼                             ▼                             ▼
┌───────────────┐           ┌───────────────┐           ┌───────────────┐
│feature-service│           │factor-service │           │ llm-service  │
│   订阅数据    │           │   订阅数据    │           │   订阅数据   │
└───────────────┘           └───────────────┘           └───────────────┘
```

**适用场景：**
- data-service → feature-service：行情数据流
- execution-service → feedback-service：订单完成事件
- risk-service → notification-service：风险告警事件

## 4.3 实时查询（HTTP / gRPC）


**适用场景：**

- 决策服务获取因子权重 → config-service

## 4.6 代码示例

```python
# 异步发布（Kafka）
from aiokafka import AIOKafkaProducer

producer = AIOKafkaProducer()
await producer.send("tradeagent.factor_data", {
    "symbol": "BTC",
    "trend_factor": 0.65,
    "sentiment_factor": 0.42
})

# 异步订阅（Kafka）
from aiokafka import AIOKafkaConsumer

consumer = AIOKafkaConsumer("tradeagent.factor_data")
async for msg in consumer:
    await process_factor_data(msg.value)

# 实时查询（HTTP）
import httpx

response = await httpx.AsyncClient().get(
    "http://state-service:8011/api/v1/state/position"
)
position = response.json()

# WebSocket 推送
from fastapi import WebSocket

async def broadcast_market(websocket: WebSocket, data: dict):
    await websocket.send_json(data)
```

## 7.1 TDP 共享库

**注意**：TDP 是**数据格式/协议**，不是独立微服务。它作为共享库被所有服务导入使用。

```
shared/tdp/
├── __init__.py
├── formatter.py       # TDPFormatter - 消息格式化
├── validator.py        # TDPValidator - 协议校验
└── schemas.py          # TDPMeta, TDPMessage 等数据类
```

**为什么不是独立服务？**
- TDP 只是格式化工具，无状态无逻辑
- 每条消息都调用，独立服务增加网络开销
- 所有服务都需遵循TDP，独立服务会造成循环依赖


## 8.1 Docker 部署 shared 共享库方案

### 方案A：Bind Mount（开发环境推荐）

```yaml
# docker-compose.yml
services:
  data-service:
    build: ./services/data-service
    volumes:
      - ./shared:/app/shared:ro    # 共享库只读挂载
    depends_on:
      - kafka
      - redis

  feature-service:
    build: ./services/feature-service
    volumes:
      - ./shared:/app/shared:ro

  factor-service:
    build: ./services/factor-service
    volumes:
      - ./shared:/app/shared:ro
```

**优点**：开发时修改 shared 代码立即生效
**缺点**：生产环境不推荐（代码分离）

### 方案B：构建时复制（生产环境推荐）

```dockerfile
# services/data-service/Dockerfile
FROM python:3.11-slim

WORKDIR /app

# 复制共享库
COPY ../shared/ ./shared/

# 安装依赖
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 复制服务代码
COPY . .

# 设置 Python 路径
ENV PYTHONPATH=/app/shared:$PYTHONPATH

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8001"]
```

**优点**：每个服务镜像独立部署
**缺点**：更新 shared 需要重新构建所有服务

### 方案C：Python Wheel 包（最佳实践）

```bash
# 打包 shared 为 wheel
cd backend/shared
pip wheel . -w dist/
```

```toml
# services/data-service/pyproject.toml
[project]
dependencies = [
    "tradeagent-shared @ file:///app/shared/dist/tradeagent_shared-1.0.0-py3-none-any.whl"
]
```

```dockerfile
# 共享 wheel 文件
COPY shared/dist/tradeagent_shared-1.0.0-py3-none-any.whl /tmp/
RUN pip install /tmp/tradeagent_shared-1.0.0-py3-none-any.whl
```

**核心功能：**

- 回撤限制
- 亏损限制
- 向 decision-service / position-service / execution-service 发送紧急指令

**控制指令类型：**
```
PAUSE_ALL:      暂停所有交易
RESUME_ALL:     恢复所有交易
EMERGENCY_EXIT: 紧急平仓所有仓位

```

**触发条件：**
```

回撤 > 10%     → 自动触发 EMERGENCY_EXIT
波动率 > 3x均值 → 触发 CIRCUIT_BREAK
连续亏损 > 5次  → 自动降低仓位 70%
```

## 8.3 各服务 Dockerfile 模板

```dockerfile
# services/<service-name>/Dockerfile
FROM python:3.11-slim

WORKDIR /app

# 安装系统依赖
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc \
    && rm -rf /var/lib/apt/lists/*

# 复制共享库
COPY ../shared/ ./shared/

# 设置 Python 路径（优先使用 shared）
ENV PYTHONPATH=/app/shared:$PYTHONPATH

# 安装 Python 依赖
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 复制服务代码
COPY . .

# 健康检查
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:${PORT:-8001}/health || exit 1

# 启动命令（PORT 环境变量指定端口）
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "${PORT:-8001}"]
```

## 8.4 部署方案对比

| 方案 | 适用场景 | 复杂度 | 优点 | 缺点 |
|------|----------|--------|------|------|
| **Bind Mount** | 开发/调试 | ⭐ | 修改立即生效 | 代码分离 |
| **构建时复制** | 小团队/服务少 | ⭐⭐ | 镜像独立 | 更新 shared 需全量重建 |
| **Wheel 包** | 大团队/服务多 | ⭐⭐⭐ | 版本可控依赖明确 | 需要发布流程 |

**推荐**：
- 开发环境：Bind Mount
- 生产环境：Wheel 包 或 构建时复制

# 🛡️ 九、设计原则

```
✅ 每个微服务职责单一
✅ 服务间通过 API 通信，不直接访问对方数据库
✅ Kafka 用于异步数据流，Redis 用于实时缓存
✅ 实盘与模拟/回测完全隔离
✅ LLM 仅辅助（爬虫、情绪分析、解释），不直接控制交易
✅ 所有交易必须经过 risk-service 验证
✅ TDP 作为共享库（shared），所有服务导入使用，不是独立服务
✅ State/Config 客户端作为共享库，避免重复代码
✅ Regime/Portfolio/Control 服务独立，增强系统可扩展性
✅ Control-service 作为系统熔断器，防止极端风险
```
