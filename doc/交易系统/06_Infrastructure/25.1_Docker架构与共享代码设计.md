## 一、部署配置结构

### 1.1 目录结构

```
backend/
├── services/                    # 微服务
│   ├── data_service/
│   │   ├── Dockerfile
│   │   └── requirements.txt
│   ├── event_service/
│   ├── fusion_service/
│   ├── llm_service/
│   └── strategy_service/
│
├── deploy/                      # 部署配置（新增）
│   ├── docker-compose.yml       # 生产环境（镜像构建，代码固定）
│   └── docker-compose.dev.yml   # 开发环境（volume 挂载，代码热更新）
│
├── shared/                      # 共享业务代码
│   ├── config/                  # 配置管理
│   ├── state/                   # 状态管理
│   ├── tdp/                     # TDP 协议
│   ├── http_client.py
│   └── llm_client.py
│
├── infrastructure/              # 基础设施（已实现延迟导入）
│   ├── logging/
│   ├── monitoring/
│   ├── database/
│   ├── cache/
│   ├── messaging/
│   └── ...
│
└── domain/                      # 领域模型
    └── event/
```

### 1.2 文件说明

| 文件 | 用途 | 构建方式 | 代码更新 |
|------|------|----------|----------|
| `docker-compose.yml` | 生产环境 | `docker-compose build` | 需要 rebuild |
| `docker-compose.dev.yml` | 开发环境 | 直接运行，无需 build | 实时生效 |

## 二、架构模式概述

### 模式 A：完全独立（Independent Services）

每个微服务作为独立 Docker 容器，共享代码通过 **volume 挂载** 共享。

```
┌─────────────────────────────────────────────────────────────────┐
│                     Docker Compose Network                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐               │
│  │data_service │  │event_service│  │fusion_service│  ...        │
│  │ (独立镜像)  │  │ (独立镜像)  │  │ (独立镜像)  │               │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘               │
│         │                │                │                      │
│         └────────────────┼────────────────┘                      │
│                          ▼                                       │
│              ┌───────────────────────┐                           │
│              │    shared/           │  ← Volume 挂载 (只读)     │
│              │    infrastructure/    │                           │
│              │    domain/           │                           │
│              └───────────────────────┘                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| 优点 | 缺点 |
|------|------|
| ✅ 服务独立部署、版本互不影响 | ❌ 镜像体积较大（每个服务都打包 shared） |
| ✅ 团队可独立开发 | ❌ volume 挂载增加复杂性 |
| ✅ 可按需扩缩容特定服务 | ❌ 依赖本地文件系统 |

### 模式 B：共享基础镜像（Base Image Sharing）

创建包含 `shared/` 和 `infrastructure/` 的 base image，所有服务基于此构建。

```
                          ┌─────────────────┐
                          │   Base Image    │
                          │  (python:3.12)  │
                          │ + shared/       │
                          │ + infrastructure/│
                          │ + 通用依赖      │
                          └────────┬────────┘
                                   │
           ┌───────────────────────┼───────────────────────┐
           ▼                       ▼                       ▼
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│  data_service    │  │  event_service   │  │  fusion_service  │
│  FROM base_image │  │  FROM base_image │  │  FROM base_image │
│  + data_collector│  │  + event_logic   │  │  + fusion_logic  │
│  + requirements  │  │  + requirements  │  │  + requirements  │
└──────────────────┘  └──────────────────┘  └──────────────────┘
```

| 优点 | 缺点 |
|------|------|
| ✅ 减少镜像重复，快速构建 | ❌ base image 更新需要 rebuild 所有服务 |
| ✅ 镜像体积小（共享层） | ❌ 服务间依赖耦合在 base image |
| ✅ 依赖一致性好 | ❌ 需要维护 base image 版本 |

### 模式 C：Monolithic（单容器）

所有服务运行在单个容器中。

```
┌─────────────────────────────────────────┐
│              Docker Container           │
├─────────────────────────────────────────┤
│                                          │
│  ┌────────────────────────────────────┐ │
│  │         Python Process              │ │
│  │                                    │ │
│  │  data_service   → Kafka Producer   │ │
│  │  event_service  → Kafka Consumer  │ │
│  │  fusion_service → Signal Engine   │ │
│  │  strategy_service→ Trading Logic  │ │
│  │                                    │ │
│  └────────────────────────────────────┘ │
│                                          │
└─────────────────────────────────────────┘
```

| 优点 | 缺点 |
|------|------|
| ✅ 简单，调试方便 | ❌ 无法独立扩缩容 |
| ✅ 启动快速（无网络延迟） | ❌ 一个服务崩溃全部崩溃 |
| ✅ 适合开发/测试环境 | ❌ 不适合生产 |

## 三、开发环境 vs 生产环境

### 3.1 对比

| 特性 | 开发环境 (dev) | 生产环境 (prod) |
|------|----------------|-----------------|
| **配置文件** | `docker-compose.dev.yml` | `docker-compose.yml` |
| **镜像构建** | ❌ 不构建，使用 `python:3.12-slim` | ✅ `docker-compose build` |
| **代码更新** | Volume 挂载，实时生效 | 需要 rebuild |
| **依赖安装** | 每次启动时安装 | 构建时安装，镜像内固化 |
| **启动速度** | 首次慢（安装依赖），后续快 | 构建慢，运行快 |
| **适用场景** | 开发调试 | 生产部署 |

### 3.2 开发环境配置 (docker-compose.dev.yml)

```yaml
services:
  data-service:
    image: python:3.12-slim
    working_dir: /app
    volumes:
      - ..:/app:ro           # 整个项目挂载到 /app
    command: >
      bash -c "pip install -r services/data_service/requirements.txt
               && python -m services.data_service.main_kafka"
```

**特点**：
- 使用 `python:3.12-slim` 基础镜像
- `..` 指向 `backend/` 目录（相对于 deploy/ 目录）
- 代码修改后**立即生效**（容器内已挂载）

### 3.3 生产环境配置 (docker-compose.yml)

```yaml
services:
  data-service:
    build:
      context: ..              # 指向 backend/ 根目录
      dockerfile: services/data_service/Dockerfile
    restart: unless-stopped
    # 无 volume 挂载，代码固定在镜像内
```

**特点**：
- 构建独立镜像
- 代码固化在镜像内
- 启动快速稳定

### 3.4 服务 Dockerfile（生产用）

```dockerfile
# services/data_service/Dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY services/data_service/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

ENV PYTHONPATH=/app
ENV PYTHONUNBUFFERED=1

CMD ["python", "-m", "services.data_service.main_kafka"]
```

## 四、共享代码组织

### 4.1 目录职责划分

| 目录 | 内容类型 | 共享方式 | 说明 |
|------|----------|----------|------|
| `shared/` | 业务工具类、配置、状态、TDP | volume 挂载 / base image | ✅ 保持共享 |
| `infrastructure/` | 日志、监控、数据库、缓存 | volume 挂载 + 延迟导入 | ✅ 保持共享 |
| `domain/` | 领域模型、事件定义 | 打包到服务镜像 | ⚠️ 按需调整 |
| `services/*/` | 业务逻辑代码 | 独立服务 | ✅ 各服务私有 |

### 4.2 infrastructure 延迟导入优化

为避免 `infrastructure/__init__.py` 导入时加载所有子模块，已实现**延迟导入**：

```python
# infrastructure/__init__.py
def __getattr__(name):
    # 只有访问时才加载对应子模块
    if name == "LoggerFactory":
        return getattr(__import__("infrastructure.logging"), "LoggerFactory")
```

**效果**：
- `import infrastructure` → 极快，不加载任何子模块
- `infrastructure.LoggerFactory` → 按需加载 logging 模块
- 可选模块导入失败 → 只在该模块被访问时失败，不阻塞启动

### 4.3 shared 代码结构

```
shared/
├── config/              # 配置管理
│   ├── defaults/        # 默认配置（business/、infrastructure/、core/）
│   ├── enums.py         # 枚举类型
│   ├── manager.py       # 配置管理器
│   └── schemas.py       # 配置 schema
│
├── state/               # 状态管理
│   ├── manager.py       # 状态管理器
│   └── types.py         # 状态类型定义
│
├── tdp/                 # TDP 协议
│   ├── client.py        # TDP 客户端
│   ├── formatter.py    # 格式化器
│   ├── types.py         # TDP 类型定义
│   └── validator.py     # 验证器
│
├── http_client.py       # HTTP 客户端
└── llm_client.py        # LLM 客户端
```

## 五、Python 包管理

### 5.1 开发环境

```yaml
# docker-compose.dev.yml
services:
  data-service:
    image: python:3.12-slim
    command: >
      bash -c "pip install --no-cache-dir -r services/data_service/requirements.txt
               && python -m services.data_service.main_kafka"
```

**流程**：
1. 启动时从 PyPI 下载安装依赖
2. 容器运行时代码通过 volume 挂载

### 5.2 生产环境

```dockerfile
# services/data_service/Dockerfile
COPY services/data_service/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
```

**流程**：
1. Docker 构建时安装依赖到镜像
2. 运行时代码和依赖都在镜像内

### 5.3 离线/内网构建方案

| 方案 | 适用场景 | 实现方式 |
|------|----------|----------|
| **私有 PyPI** | 内网环境 | 搭建 pypiserver 或 artifice |
| **预下载 wheel** | 少量依赖 | `pip download -r requirements.txt` |
| **多阶段构建** | 完全离线 | 在有网环境打包，导入镜像 |

## 六、快速启动命令

### 6.1 生产环境

```bash
# 进入部署配置目录
cd /Users/yangdezeng/00_crypto/00_trade_agent/20260506/backend/deploy

# 构建所有服务镜像
docker-compose build

# 启动所有服务
docker-compose up -d

# 查看服务状态
docker-compose ps

# 查看特定服务日志
docker-compose logs -f data-service

# 重新构建并启动单个服务
docker-compose build data-service
docker-compose up -d data-service

# 停止所有服务
docker-compose down
```

### 6.2 开发环境

```bash
# 进入部署配置目录
cd /Users/yangdezeng/00_crypto/00_trade_agent/20260506/backend/deploy

# 启动开发环境（无需 build）
docker-compose -f docker-compose.dev.yml up -d

# 查看服务日志
docker-compose -f docker-compose.dev.yml logs -f data-service

# 停止开发环境
docker-compose -f docker-compose.dev.yml down
```

### 6.3 代码修改后

| 环境 | 代码修改后 |
|------|-----------|
| **开发环境** | 无需任何操作，代码已挂载，修改立即生效 |
| **生产环境** | `docker-compose build <service> && docker-compose up -d <service>` |

### 6.4 环境切换

```bash
# 使用 docker-compose -f 指定文件
docker-compose -f docker-compose.dev.yml ps   # 查看开发环境
docker-compose -f docker-compose.yml ps       # 查看生产环境

# 或者创建别名（添加到 ~/.bashrc 或 ~/.zshrc）
alias dev-up="docker-compose -f /path/to/docker-compose.dev.yml up -d"
alias prod-up="docker-compose -f /path/to/docker-compose.yml up -d"
```

## 七、架构选择建议

| 场景 | 推荐模式 | 理由 |
|------|----------|------|
| **开发/调试** | docker-compose.dev.yml | 代码热更新，无需 rebuild |
| **团队独立开发** | A (完全独立) | 服务解耦，独立部署 |
| **生产环境（推荐）** | docker-compose.yml + base image | 镜像复用，快速部署 |
| **大型团队** | B + K8s | 可扩缩容，版本管理 |
