# TradeAgent Docker 部署与 CI/CD

## 一、部署架构概览

TradeAgent 采用容器化部署方案，通过 GitHub Actions 实现自动化 CI/CD 流程。

| 组件 | 技术方案 |
|------|----------|
| **容器化** | Docker + Docker Compose |
| **后端镜像仓库** | GitHub Container Registry (ghcr.io) |
| **前端部署** | Vercel (可选) |
| **CI/CD** | GitHub Actions |
| **基础设施** | Kafka + Redis |

## 三、Dockerfile 设计

### 3.1 后端服务 Dockerfile

以 [data-service Dockerfile](file:///Users/yangdezeng/00_crypto/00_trade_agent/20260506/backend/services/data_service/Dockerfile) 为例：

```dockerfile
FROM python:3.12-slim

WORKDIR /app

# 先复制依赖，利用 Docker 缓存
COPY services/data_service/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 再复制代码
COPY . .

ENV PYTHONPATH=/app
ENV PYTHONUNBUFFERED=1

CMD ["python", "-m", "services.data_service.main_kafka"]
```

### 3.2 镜像构建优化

- **分层构建**：依赖层和代码层分离，提高缓存命中率
- **slim 镜像**：使用 python:3.12-slim 减少镜像体积
- **no-cache-dir**：避免 pip 缓存，减小镜像
- **PYTHONUNBUFFERED=1**：日志即时输出

## 五、CI/CD 自动化流程

### 5.1 后端 CI/CD (GitHub Actions)

工作流文件：[.github/workflows/backend.yml](file:///Users/yangdezeng/00_crypto/00_trade_agent/20260506/.github/workflows/backend.yml)

**触发条件：**
- push 到 main 分支，且 `backend/**` 目录有变更
- 手动触发 (workflow_dispatch)

**流程步骤：**

```
代码 Push
    │
    ▼
Checkout 代码
    │
    ▼
设置 Docker Buildx
    │
    ▼
登录 ghcr.io (使用 GITHUB_TOKEN)
    │
    ▼
提取元数据 (标签)
    │
    ├─▶ 构建 data-service ──┐
    ├─▶ 构建 event-service ──┤
    ├─▶ 构建 fusion-service ─┤
    ├─▶ 构建 strategy-service┤
    └─▶ 构建 llm-service ────┤
                             │
                             ▼
                      Push 到 ghcr.io
                         - {sha} 标签
                         - latest 标签
```

**镜像标签策略：**
- `{branch}-{sha}`：精确的提交版本
- `latest`：默认分支最新版本

### 5.2 前端 CI/CD (GitHub Actions)

工作流文件：[.github/workflows/frontend.yml](file:///Users/yangdezeng/00_crypto/00_trade_agent/20260506/.github/workflows/frontend.yml)

**触发条件：**
- push 到 main 分支，且 `frontend/**` 目录有变更
- 手动触发 (workflow_dispatch)

**流程步骤：**

```
代码 Push
    │
    ▼
Checkout 代码
    │
    ▼
Setup Node.js (v20)
    │
    ▼
npm ci (安装依赖)
    │
    ▼
Pull Vercel 配置
    │
    ▼
Vercel Build (生产构建)
    │
    ▼
Vercel Deploy (生产部署)
```

## 六、GitHub Secrets 配置

### 6.1 必需的 Secrets

| Secret 名称 | 说明 | 获取方式 |
|-------------|------|----------|
| **VERCEL_TOKEN** | Vercel API Token | https://vercel.com/account/tokens |
| **VERCEL_ORG_ID** | Vercel 组织 ID | Vercel 项目设置 |
| **VERCEL_PROJECT_ID** | Vercel 项目 ID | Vercel 项目设置 |

**注意：** 后端 CI/CD 不需要额外 Secrets，使用内置的 `GITHUB_TOKEN` 即可。

### 6.2 配置步骤

1. 进入 GitHub 仓库：https://github.com/ydz-auto/trade_sys
2. 导航到 **Settings** > **Secrets and variables** > **Actions**
3. 点击 **New repository secret**
4. 添加上述 Secrets (如果需要前端部署)

## 七、生产环境部署指南

### 7.1 前置条件

- 服务器已安装 Docker 和 Docker Compose
- 服务器可访问 GitHub Container Registry
- 已配置 Docker 镜像加速 (可选但推荐)

### 7.2 首次部署

**步骤 1：登录 ghcr.io**
```bash
# 生成 GitHub Personal Access Token (PAT)
# 访问：https://github.com/settings/tokens
# 需要权限：read:packages, write:packages

docker login ghcr.io -u ydz-auto -p YOUR_GITHUB_PAT
```

**步骤 2：拉取代码和部署**
```bash
cd /path/to/deploy
git clone https://github.com/ydz-auto/trade_sys.git
cd trade_sys/backend/deploy

# 拉取最新镜像
docker-compose -f docker-compose.prod.yml pull

# 启动所有服务
docker-compose -f docker-compose.prod.yml up -d
```

**步骤 3：验证部署**
```bash
# 查看服务状态
docker-compose -f docker-compose.prod.yml ps

# 查看日志
docker-compose -f docker-compose.prod.yml logs -f

# 访问 Kafka UI
# http://your-server-ip:8080
```

### 7.3 更新部署

当 CI/CD 完成新镜像构建后：

```bash
cd /path/to/trade_sys/backend/deploy

# 拉取最新镜像
docker-compose -f docker-compose.prod.yml pull

# 重启服务
docker-compose -f docker-compose.prod.yml up -d
```

## 八、运维操作

### 8.1 常用命令

```bash
# 查看服务状态
docker-compose -f docker-compose.prod.yml ps

# 查看所有服务日志
docker-compose -f docker-compose.prod.yml logs -f

# 查看特定服务日志
docker-compose -f docker-compose.prod.yml logs -f data-service
docker-compose -f docker-compose.prod.yml logs -f event-service

# 重启特定服务
docker-compose -f docker-compose.prod.yml restart data-service

# 停止所有服务
docker-compose -f docker-compose.prod.yml down

# 停止并删除数据卷 (谨慎使用!)
docker-compose -f docker-compose.prod.yml down -v
```

### 8.2 数据备份

**备份 Kafka 数据：**
```bash
# Kafka 数据已通过 volume 持久化
# 定期备份 volume 内容
docker run --rm -v trade_sys_kafka_data:/data -v $(pwd):/backup alpine tar czf /backup/kafka-backup-$(date +%Y%m%d).tar.gz /data
```

**备份 Redis 数据：**
```bash
# Redis 数据已通过 volume 持久化
docker run --rm -v trade_sys_redis_data:/data -v $(pwd):/backup alpine tar czf /backup/redis-backup-$(date +%Y%m%d).tar.gz /data
```

### 8.3 监控和告警

- **Kafka UI**：http://your-server:8080 - 监控消息流和消费者状态
- **Docker 健康检查**：各服务配置了健康检查，可通过 Docker API 获取状态
- **日志监控**：可集成 ELK 或 Loki 进行日志聚合和告警

## 九、Docker 镜像加速配置 (中国地区)

### 9.1 配置 Docker Daemon

编辑 `/etc/docker/daemon.json`：
```json
{
  "registry-mirrors": [
    "https://docker.1ms.run",
    "https://dockerproxy.link",
    "https://docker.m.daocloud.io",
    "https://registry.cyou"
  ]
}
```

重启 Docker：
```bash
sudo systemctl daemon-reload
sudo systemctl restart docker
```

## 十一、故障排查

### 11.1 常见问题

**Q: 服务无法连接 Kafka**
- 检查 kafka 容器健康状态：`docker ps | grep kafka`
- 查看 kafka 日志：`docker-compose -f docker-compose.prod.yml logs kafka`
- 确认网络连接正常

**Q: 拉取镜像失败**
- 确认已正确登录 ghcr.io
- 检查 GitHub PAT 是否有效且有 read:packages 权限
- 配置镜像加速 (如在中国)

**Q: 服务反复重启**
- 查看服务日志：`docker-compose -f docker-compose.prod.yml logs <service-name>`
- 检查环境变量配置是否正确
- 确认 Kafka/Redis 是否正常运行

### 11.2 日志位置

所有服务日志可通过 Docker 查看：
```bash
# 所有服务
docker-compose -f docker-compose.prod.yml logs -f

# 特定服务
docker-compose -f docker-compose.prod.yml logs -f data-service
```
