# TradeAgent 中间件共享架构设计文档

---

# 🧠 一、文档定位

本文档定义 TradeAgent 系统中 **中间件的共享架构**，明确：
- 各中间件的用途和选型
- 共用 vs 独占的使用模式
- 依赖服务列表
- 配置规范和资源规划

> **设计原则**：中间件作为基础设施层，被多个微服务共用，通过统一配置管理实现服务解耦。

---

# 📊 二、中间件总览

| 中间件 | 用途 | 模式 | 依赖服务 |
|--------|------|------|----------|
| **Kafka** | 异步事件流 | ✅ 共用 | 所有服务 |
| **Redis** | 缓存 / Session | ✅ 共用 | 所有服务 |
| **PostgreSQL** | 关系数据存储 | ✅ 共用 | auth / risk / decision / position / execution / state / config |
| **ClickHouse** | 分析数据存储 | ✅ 共用 | data / feature / factor / feedback |
| **Nginx** | 反向代理 | ⚠️ 独占 | api-gateway |
| **Prometheus** | 指标采集 | ✅ 共用 | 所有服务 |
| **ELK** | 日志采集 | ✅ 共用 | 所有服务 |

# 🔄 三、Kafka（异步事件流）

## 3.1 定位

Kafka 作为系统的 **异步消息中枢**，承担：
- 服务间事件传递
- 数据流管道
- 订单/信号事件广播

## 3.2 Topic 设计

```
tradeagent/
├── raw_data                    # 原始行情数据
├── feature_data               # 特征计算结果
├── factor_data                # 因子计算结果
├── risk_alerts                # 风控告警事件
├── trading_signals            # 交易信号事件
├── order_events               # 订单状态变更
├── execution_results          # 执行结果
└── feedback_data              # 反馈闭环数据
```

## 3.3 服务依赖

| 服务 | 角色 | Topic |
|------|------|-------|
| data-service | Producer | raw_data |
| feature-service | Consumer | raw_data → Producer: feature_data |
| factor-service | Consumer | feature_data |
| risk-service | Consumer | factor_data, order_events |
| decision-service | Producer | trading_signals |
| execution-service | Consumer/Producer | trading_signals → order_events |
| feedback-service | Consumer | order_events, execution_results |
| notification-service | Consumer | risk_alerts |

## 3.4 配置示例

```yaml
# docker-compose.yml
kafka:
  image: confluentinc/cp-kafka:7.5.0
  ports:
    - "9092:9092"
  environment:
    KAFKA_BROKER_ID: 1
    KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
    KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092
    KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
    KAFKA_MAX_PARTITION_BYTES: 1073741824  # 1GB
    KAFKA_RETENTION_MS: 604800000           # 7天
```

# 💾 四、Redis（缓存 / Session）

## 4.1 定位

Redis 作为系统的 **高性能缓存和实时状态存储**：
- 热点数据缓存
- Session 管理
- 实时状态存储（仓位、账户）
- 分布式锁
- 消息 Pub/Sub

## 4.2 Key 设计规范

```
tradeagent:{environment}:{service}:{resource}

# 示例
tradeagent:live:state:positions:BTC
tradeagent:live:risk:index
tradeagent:live:config:factor_weights
tradeagent:backtest:cache:features:BTC:1h
```

## 4.3 数据库分片

| DB | 用途 | TTL |
|----|------|-----|
| DB 0 | 实时状态（仓位/账户） | 无 |
| DB 1 | 缓存数据 | 5min-1h |
| DB 2 | Session | 24h |
| DB 3 | 分布式锁 | 30s |
| DB 4 | Pub/Sub | 无 |

## 4.4 配置示例

```yaml
# docker-compose.yml
redis:
  image: redis:7-alpine
  ports:
    - "6379:6379"
  command: redis-server --maxmemory 2gb --maxmemory-policy allkeys-lru
```

## 4.5 服务依赖

```
✅ 所有服务共用 Redis
  - state-service: 实时状态存储
  - config-service: 配置缓存
  - risk-service: 风险指数缓存
  - decision-service: 决策上下文缓存
  - execution-service: 订单状态缓存
  - auth-service: Session 存储
```

- 用户账户数据

- 审计日志

| 数据库 | 服务 | 用途 |
|--------|------|------|

| tradeagent_risk | risk-service | 风控规则、风险记录 |
| tradeagent_decision | decision-service | 决策记录、信号历史 |
| tradeagent_position | position-service | 仓位历史 |
| tradeagent_execution | execution-service | 订单记录 |

| tradeagent_config | config-service | 配置版本管理 |

## 5.3 连接池配置

```yaml
# PostgreSQL 全局配置

  image: postgres:15-alpine
  environment:
    POSTGRES_MAX_CONNECTIONS: 200
  # 各服务连接数规划
  # auth-service: 20
  # risk-service: 30
  # decision-service: 30
  # position-service: 25
  # execution-service: 25
  # state-service: 30
  # config-service: 20
  # 总计: 180 (预留20给管理操作)
```

# 🌐 七、Nginx（反向代理）

## 7.1 定位

Nginx 作为系统的 **唯一入口**：
- 反向代理到 api-gateway
- 负载均衡（多实例部署）
- SSL 终止
- 静态资源服务

## 7.2 架构

```
Client → Nginx (443) → api-gateway (8000)
```

## 7.3 配置示例

```nginx
# nginx.conf
upstream api_gateway {
    least_conn;
    server api-gateway-1:8000;
    server api-gateway-2:8000;
}

server {
    listen 443 ssl;
    server_name tradeagent.example.com;

    ssl_certificate /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;

    location / {
        proxy_pass http://api_gateway;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

## 7.4 注意

> ⚠️ **Nginx 仅限 api-gateway 独占使用**，不与其他服务共享。

# 📊 八、Prometheus（指标采集）

## 8.1 定位

Prometheus 作为系统的 **指标采集中枢**：
- 服务健康指标
- 业务指标（延迟、吞吐量）
- 资源指标（CPU、内存）

## 8.2 服务端口映射

| 服务 | 端口 | 指标路径 |
|------|------|----------|
| data-service | 8001 | /metrics |
| feature-service | 8002 | /metrics |
| factor-service | 8003 | /metrics |
| risk-service | 8004 | /metrics |
| decision-service | 8005 | /metrics |
| position-service | 8006 | /metrics |
| execution-service | 8007 | /metrics |
| state-service | 8011 | /metrics |
| config-service | 8012 | /metrics |

## 8.3 配置示例

```yaml
prometheus:
  image: prom/prometheus:latest
  ports:
    - "9090:9090"
  volumes:
    - ./prometheus.yml:/etc/prometheus/prometheus.yml
```

# 📋 九、ELK（日志采集）

## 9.1 定位

ELK 作为系统的 **日志中枢**：
- 统一日志收集
- 日志搜索分析
- 日志可视化

## 9.2 组件

| 组件 | 用途 |
|------|------|
| Elasticsearch | 日志存储和搜索 |
| Logstash | 日志收集和处理 |
| Kibana | 日志可视化 |

## 9.3 服务日志规范

所有服务使用 **JSON 格式** 输出日志：

```json
{
    "timestamp": "2024-01-01T12:00:00.000Z",
    "level": "INFO",
    "service": "decision-service",
    "message": "Signal generated",
    "request_id": "req_12345",
    "extra": {}
}
```

## 9.4 配置示例

```yaml
# docker-compose.yml
elasticsearch:
  image: elasticsearch:8.11.0
  ports:
    - "9200:9200"

logstash:
  image: logstash:8.11.0
  volumes:
    - ./logstash.conf:/usr/share/logstash/pipeline/logstash.conf

kibana:
  image: kibana:8.11.0
  ports:
    - "5601:5601"
```

# ✅ 十二、设计原则

```
✅ 中间件共用，数据库按服务隔离
✅ Redis 作为统一缓存层，减少数据库压力
✅ Kafka 作为异步中枢，解耦服务间调用
✅ Nginx 统一入口，实现安全防护
✅ Prometheus + ELK 实现可观测性全覆盖
✅ 所有中间件配置版本化管理
```
