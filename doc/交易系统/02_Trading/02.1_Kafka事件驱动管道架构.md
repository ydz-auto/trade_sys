# TradeAgent Kafka 事件驱动管道架构

## 一、架构概览

TradeAgent 采用 **Kafka 事件驱动架构**，实现了一个可扩展、高解耦的微服务系统。整个管道由 4 个核心服务组成，通过 Kafka Topic 串联，形成完整的数据处理流程。

### 核心组件关系图

```
外部数据 (新闻、行情、ETF)
    ↓
┌─────────────────────────────────────────────────────────────────┐
│  data_service              (数据采集 + 消息生产者)             │
│  - 新闻采集                 (CoinDesk, CoinTelegraph 等)       │
│  - 行情数据采集             (Binance, OKX 等)                 │
│  - ETF 数据采集                                             │
└────────────┬────────────────────────────────────────────────┘
             │
             ▼ Kafka Topic: tradeagent.raw_data
┌─────────────────────────────────────────────────────────────────┐
│  event_service              (事件检测 + 消费者 + 生产者)         │
│  - 关键词匹配检测事件          (inflow, hack, depeg 等)        │
│  - 转换为标准化 Event 模型                                       │
└────────────┬────────────────────────────────────────────────┘
             │
             ▼ Kafka Topic: tradeagent.events
┌─────────────────────────────────────────────────────────────────┐
│  fusion_service             (信号融合 + 消费者 + 生产者)         │
│  - 时间窗口缓冲                (默认 300s)                      │
│  - 事件聚合和冲突解决                                           │
│  - 置信度计算                                                  │
└────────────┬────────────────────────────────────────────────┘
             │
             ▼ Kafka Topic: tradeagent.signals
┌─────────────────────────────────────────────────────────────────┐
│  strategy_service           (交易决策 + 消费者)                 │
│  - 根据信号决定交易动作          (LONG/SHORT/HOLD)               │
│  - 计算仓位大小                                                  │
│  - 输出最终决策                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 二、核心微服务详解

### 1. data_service (数据采集服务)

**职责：**
- 从外部数据源（新闻、行情、ETF）采集原始数据
- 将数据标准化为 RawData 消息体
- 发布到 Kafka Topic: `tradeagent.raw_data`

**核心代码文件：**
- [main_kafka.py](file:///Users/yangdezeng/00_crypto/00_trade_agent/20260506/backend/services/data_service/main_kafka.py)

**采集源：**
- CoinDesk RSS
- CoinTelegraph RSS
- CryptoPanic
- 中文新闻源 (JinShi, ODaily 等)

**消息格式 (RawData)：**
```python
{
    "type": "news",           // 数据类型: news, market, etf
    "symbol": "BTC",          // 影响的币种
    "source": "coindesk",     // 数据源
    "data": { ... },          // 原始数据
    "timestamp": 1710000000   // 时间戳
}
```

---

### 2. event_service (事件检测服务)

**职责：**
- 消费 `tradeagent.raw_data` Topic
- 使用关键词匹配检测事件
- 转换为标准化的 Event 模型
- 发布到 `tradeagent.events` Topic

**核心代码文件：**
- [main_kafka.py](file:///Users/yangdezeng/00_crypto/00_trade_agent/20260506/backend/services/event_service/main_kafka.py)

**事件类型：**
| 关键词 | 事件类型 | 方向 | 说明 |
|--------|----------|------|------|
| inflow | `FLOW_ETF_INFLOW` | bullish | ETF 资金流入 |
| outflow | `FLOW_ETF_OUTFLOW` | bearish | ETF 资金流出 |
| etf | `POLICY_ETF_APPROVAL` | bullish | ETF 审批相关 |
| hack | `PROTOCOL_HACK` | bearish | 协议被攻击 |
| exploit | `PROTOCOL_HACK` | bearish | 漏洞利用 |
| depeg | `RISK_STABLECOIN_DEPEG` | bearish | 稳定币脱钩 |
| institutional | `POLICY_REGULATION_POSITIVE` | bullish | 机构采用 |
| adoption | `POLICY_REGULATION_POSITIVE` | bullish | 监管利好 |

**消息格式 (Event)：**
```python
{
    "event_type": "FLOW_ETF_INFLOW", // 事件类型
    "category": "flow",              // 分类
    "asset": "BTC",                  // 交易品种
    "direction": "bullish",          // 方向
    "strength": 0.85,                // 强度 (0-1)
    "sources": ["news"],             // 来源
    "timestamp": 1710000000,         // 时间戳
    "metadata": { ... }              // 元数据
}
```

---

### 3. fusion_service (信号融合服务)

**职责：**
- 消费 `tradeagent.events` Topic
- 时间窗口缓冲 (默认 300s)
- 事件聚合和冲突解决
- 计算信号置信度
- 发布融合后的 Signal 到 `tradeagent.signals`

**核心代码文件：**
- [main_kafka.py](file:///Users/yangdezeng/00_crypto/00_trade_agent/20260506/backend/services/fusion_service/main_kafka.py)
- [engine.py](file:///Users/yangdezeng/00_crypto/00_trade_agent/20260506/backend/services/fusion_service/engine.py)

**关键特性：**
- **冷却机制：** 10s 内不重复生成信号
- **冲突解决：** 同一资产的 bullish/bearish 信号相互抵消
- **信号聚合：** 同一方向的多个事件，置信度累加

**消息格式 (Signal)：**
```python
{
    "signal": "BTC_BULLISH",        // 信号标识
    "direction": "bullish",         // 方向
    "confidence": 0.72,             // 置信度 (0-1)
    "consensus": 0.65,              // 共识度
    "event_types": [...],           // 涉及的事件类型
    "assets": ["BTC"],              // 涉及的资产
    "event_count": 3,               // 事件数量
    "source": "fusion_service"      // 来源
}
```

---

### 4. strategy_service (交易决策服务)

**职责：**
- 消费 `tradeagent.signals` Topic
- 根据信号决定最终交易动作
- 计算仓位大小
- 输出最终交易决策

**核心代码文件：**
- [main_kafka.py](file:///Users/yangdezeng/00_crypto/00_trade_agent/20260506/backend/services/strategy_service/main_kafka.py)

**决策规则：**
| 条件 | 动作 | 仓位计算 |
|------|------|----------|
| 置信度 < 0.1 | HOLD | 0.0 |
| direction == bullish | LONG | min(置信度 × 0.8, 0.9) |
| direction == bearish | SHORT | min(置信度 × 0.8, 0.9) |

**决策输出示例：**
```
🎯 FINAL TRADING DECISION
  Symbol:     BTC
  Signal:     BTC_BULLISH
  Confidence: 0.72
  Consensus:  0.65
--------------------------------
  ✅ ACTION:   LONG
  📊 POSITION: 0.58
  💡 REASON:   置信度 0.72
```

## 四、基础设施服务

### Kafka + Zookeeper
- **Broker:** localhost:9092
- **内部通信:** kafka:29092 (Docker 网络)
- **Zookeeper:** localhost:2181

### Redis
- **地址:** localhost:6379
- **用途:** 缓存、状态存储、消息队列

### Kafka UI
- **地址:** http://localhost:8080
- **用途:** Kafka 监控、Topic 管理、消息查看

## 五、部署架构

### Docker Compose 配置

完整配置见 [docker-compose.dev.yml](file:///Users/yangdezeng/00_crypto/00_trade_agent/20260506/backend/deploy/docker-compose.dev.yml)

**服务启动顺序：**
1. Zookeeper
2. Kafka (等待 Zookeeper)
3. Redis
4. Kafka UI (等待 Kafka)
5. data_service (等待 Kafka 健康检查)
6. event_service (等待 data_service)
7. fusion_service (等待 event_service)
8. strategy_service (等待 fusion_service)

**网络配置：**
- 所有服务在 `kafka-network` 网络内通信
- 外部访问通过端口映射

## 六、快速启动指南

### 方式一：Docker Compose (推荐)

```bash
cd backend/deploy
docker-compose -f docker-compose.dev.yml up -d
```

**查看服务状态：**
```bash
docker-compose -f docker-compose.dev.yml ps
```

**查看日志：**
```bash
docker-compose -f docker-compose.dev.yml logs -f data-service
docker-compose -f docker-compose.dev.yml logs -f event-service
docker-compose -f docker-compose.dev.yml logs -f fusion-service
docker-compose -f docker-compose.dev.yml logs -f strategy-service
```

**停止服务：**
```bash
docker-compose -f docker-compose.dev.yml down
```

---

### 方式二：本地 Python 启动

**前提条件：**
- Kafka 运行在 localhost:9092
- Redis 运行在 localhost:6379

**步骤 1：启动基础设施**
```bash
cd backend/deploy
docker-compose -f docker-compose.dev.yml up -d zookeeper kafka redis kafka-ui
```

**步骤 2：安装依赖**
```bash
cd backend
pip install -r services/data_service/requirements.txt
pip install -r services/event_service/requirements.txt
pip install -r services/fusion_service/requirements.txt
pip install -r services/strategy_service/requirements.txt
```

**步骤 3：打开 4 个终端分别启动服务**

**终端 1 - fusion_service：**
```bash
cd backend
python -m services.fusion_service.main_kafka
```

**终端 2 - strategy_service：**
```bash
cd backend
python -m services.strategy_service.main_kafka
```

**终端 3 - event_service：**
```bash
cd backend
python -m services.event_service.main_kafka
```

**终端 4 - data_service：**
```bash
cd backend
python -m services.data_service.main_kafka
```

## 八、架构优势

1. **高解耦：** 服务间通过 Kafka 通信，互不依赖
2. **可扩展：** 可独立扩展每个服务
3. **容错：** Kafka 持久化消息，服务重启不丢失数据
4. **可观测：** 通过 Kafka UI 实时监控数据流动
5. **异步处理：** 事件驱动，响应式架构
